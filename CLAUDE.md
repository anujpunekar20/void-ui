# void-ui

Dark-first, brutalist React UI component library. TypeScript + CSS Custom Properties + StyleX. Zero runtime styling dependencies beyond StyleX's tiny props-merger.

## Project Structure

```
src/
  components/     # One folder per component (Component.tsx, index.ts)
  styles/
    tokens.css        # All CSS custom properties (design tokens) — the actual values, and the public theming contract consumers override
    tokens.stylex.ts  # stylex.defineConsts() typed accessors (tokens.accent, tokens.spaceMd, ...) for tokens.css — compile-time typo safety, zero runtime/CSS difference
    index.css     # Imports Google Fonts + tokens.css; loaded by Storybook & consumers
  index.ts        # Main entry — re-exports everything
stories/          # Storybook stories per component
docs/superpowers/ # Design specs and implementation plans
```

## Commands

```bash
npm run dev       # Start Storybook (localhost:6006) — local live preview only
npm run build     # Bundle with Vite library mode (ESM + CJS + types)
npm run typecheck # tsc --noEmit
npm run lint      # ESLint
npm test          # Vitest
```

## Design Tokens

All colors, spacing, radii, and animation durations live in `src/styles/tokens.css` as CSS custom properties prefixed `--void-*`. Never hardcode values — always reference a token via `src/styles/tokens.stylex.ts`'s `tokens` object (e.g. `tokens.accent`, not the raw string `'var(--void-accent)'`) — `stylex.defineConsts()` catches a typo'd token name at compile time; a raw string typo fails silently. `defineConsts` just inlines the given value as-is (here, the exact `var(--void-*)` string) — it does **not** create new CSS variables or rename anything, so `tokens.css` stays the single source of truth for actual values and consumer theming is unaffected. For a composite CSS value (multiple tokens or extra text in one property, e.g. `transition`), use a template literal: `` `border-color ${tokens.durationBase} ${tokens.easeOut}` `` — a token can only be used bare (`tokens.accent`, no quotes) when it's the *entire* value.

### Key tokens
- `--void-radius: 0px` — strict sharp edges (brutalist aesthetic, no rounded corners anywhere)
- `--void-font-mono: 'Kode Mono', monospace` — the single typeface for the entire system
- `--void-accent: #c160ef` — violet, used for interactive states, focus rings, active indicators
- `--void-shadow: #3a3a3a` — hard-edged, zero-blur offset shadow (`Npx Npx 0 0`, never blurred) for raised/floating surfaces (`Button`, `Card`, `Dropdown`'s menu, `Tooltip`'s bubble, `Select`'s listbox); a blurred shadow would read as glassmorphism and break the brutalist language, so blur radius is always `0`
- Spacing: `--void-space-xs/sm/md/lg/xl` (4/8/16/24/48px)
- Font sizes: `--void-font-size-display` through `--void-font-size-label-sm` (48px → 12px)

## Component Conventions

- One component per folder: `src/components/Button/`
- Each folder: `Button.tsx`, `index.ts` (no separate CSS file — styles live in Button.tsx via StyleX)
- Props interface exported and named `{Component}Props`
- Variants and sizes via a `variant` / `size` prop — not className overrides
- StyleX `stylex.create()` for all styles — reference tokens via the `tokens` object from `src/styles/tokens.stylex.ts` (see Design Tokens section), not raw `var(--void-*)` strings

## Styling Rules

- **StyleX only** — compile-time atomic CSS, not runtime CSS-in-JS
- No inline styles
- No plain CSS files per component (tokens.css is the exception)
- No Tailwind
- CSS Custom Properties only for theming — consumed inside StyleX style objects
- All interactive states: hover, focus-visible, active, disabled

## Accessibility

- All interactive components must have keyboard support
- Use semantic HTML — `<button>`, `<input>`, `<dialog>`, etc.
- ARIA attributes where native semantics fall short
- Focus-visible ring uses `--void-accent` with `--void-accent-glow`

## Publishing

- Package: `@anuj20/void-ui` on npmjs.com (scoped — the unscoped `void-ui` name was already taken by an unrelated Vue library)
- `publishConfig.access: "public"` in `package.json` — required for scoped packages to publish publicly, otherwise npm defaults them to private
- Peer deps: `react`, `react-dom` (not bundled)
- Consumers import styles once: `import '@anuj20/void-ui/styles'`
- Bump version with `npm version patch|minor|major` before publish
- Publishing is manual (`npm publish`) — no CI auto-publish-on-tag currently; `prepublishOnly` runs the build automatically so a stale `dist/` can never ship
- Repo: `github.com/anujpunekar20/void-ui`; Storybook auto-deploys to GitHub Pages (`https://anujpunekar20.github.io/void-ui/`) via `.github/workflows/deploy-storybook.yml` on every push to `main`

## Build Output

`npm run build` runs `vite build` then merges two CSS files into `dist/index.css`:
- `dist/style.css` — CSS custom property tokens + Google Fonts import (from `src/styles/index.css`)
- `dist/assets/stylex.css` — atomic CSS classes generated by StyleX from `stylex.create()` calls

Consumers import both via a single `import '@anuj20/void-ui/styles'` (maps to `dist/index.css`).

## Implemented Components

- **Button** — `variant: solid | outline | ghost | destructive`, `size: sm | md | lg`; wrapped in `forwardRef<HTMLButtonElement, ButtonProps>` so a consumer (or a component like `Dropdown`) attaching a ref to it actually reaches the underlying `<button>` DOM node, instead of React silently dropping the ref (a plain function component can't receive one without `forwardRef`)
- **Badge** — `variant: default | accent | outline | destructive`
- **Input** — `label?`, `error?`, all standard input attrs; auto-generates accessible `id` via `useId()`
- **Textarea** — `label?`, `error?`, all standard textarea attrs; same pattern as Input
- **Card** — `header?`, `children`; bordered container, no shadows
- **Spinner** — `size: sm | md | lg`, `aria-label?`; respects `prefers-reduced-motion`
- **Select** — free-text combobox (not a strict native `<select>`), datalist-style: input + filtered, keyboard-navigable suggestions dropdown. Accepts either `options: {value,label}[]` or `<Option>` children (children take precedence). Full ARIA combobox pattern (`role="combobox"`, `aria-expanded`, `aria-activedescendant`, `role="listbox"`/`role="option"`)
- **Option** — marker component for `Select` children (`value`, `label?`, `disabled?`); Select reads its props rather than rendering it directly, so it's Select-only, not standalone-renderable
- **Checkbox** — `label?`, `error?`, all standard checkbox input attrs; styled via `appearance: none` on the native input + `:checked`/`:focus-visible` pseudo-classes (no JS state tracking — CSS-only, unlike Select)
- **Toggle** — `label?`, `error?`, all standard checkbox input attrs (semantically `<input type="checkbox" role="switch">`); rectangular track (0px radius, no exceptions) with a sliding thumb drawn via `::after`, position toggled by conditionally merging a `trackChecked` style object (mirrors `checked`/`defaultChecked`/controlled-vs-uncontrolled via local state, same pattern as `Select`'s `query`)
- **Dropdown** — `trigger: ReactElement` (any element — native tag or a `forwardRef`-wrapped component), `items: {value, label, disabled?, destructive?, onSelect?}[]`, `placement?: Placement` (any floating-ui placement — `top`/`bottom`/`left`/`right` × `-start`/`-end`, default `bottom-start`; `flip`/`shift` middleware auto-adjusts if there's no room; `offset(MENU_GAP)` = 6px gap). First component built on `@floating-ui/react` — `useFloating` + `offset/flip/shift/autoUpdate` for positioning, `useClick`/`useDismiss`/`useRole('menu')`/`useListNavigation` for interaction (arrow-key nav, `Escape`/outside-click dismiss), `FloatingFocusManager` for focus trap/return, **`FloatingPortal`** wrapping the menu (renders into `document.body`, avoiding stacking-context bugs). Position is measured off a plain `<span>` wrapper around the trigger (`refs.setReference`), not the trigger itself — `cloneElement` puts `aria-haspopup`/`aria-expanded`/click handling on the real trigger element (correct a11y semantics) but deliberately leaves its `ref` untouched. This split matters: if the trigger has its own press/active CSS transform (e.g. `Button`'s `:active { transform: scale(0.97) }`), measuring `getBoundingClientRect()` directly on it would pick up that transform and jitter the menu's computed position on every press — the wrapper span never transforms, so the measurement stays stable. Note: a *custom* trigger component (not a native DOM tag) must still be `forwardRef`-wrapped for its own ref-consumers, though `Dropdown` itself no longer depends on that for positioning; `Button` was converted to `forwardRef` regardless, since it's still needed for a11y/interaction props to land correctly and for consumers who ref it directly. `Popover` (deferred — Modal may replace it) and `Tooltip` reuse the same floating-ui hooks and the anchor-span pattern
- **Tooltip** — `trigger: ReactElement`, `label: string` (short text only — tooltips must stay non-interactive, so this is deliberately not arbitrary `children`), `placement?: Placement` (default `'top'`, unlike Dropdown's `'bottom-start'`). Same foundation as `Dropdown` (`useFloating`, the anchor-`<span>` measurement wrapper, `FloatingPortal`), but differs everywhere the trigger model differs: opens via `useHover` (`delay: { open: 200, close: 0 }`, mirrors `--void-duration-base`) + `useFocus` (`:focus-visible`-only by default — keyboard nav shows it, mouse-click-to-focus doesn't; **note**: `happy-dom` doesn't implement `:focus-visible` matching, `matches(':focus-visible')` always returns `false`, so this specific open path can't be exercised in this project's tests — see the comment in `Tooltip.test.tsx`), `useRole('tooltip')`, and **no `FloatingFocusManager`** — a tooltip must never steal or trap focus. Renders a visible arrow via floating-ui's `arrow()` middleware + `<FloatingArrow>` (its `tipRadius` already defaults to `0` — sharp point, fits the brutalist aesthetic with no extra styling needed)

## Stack

- React 18+ / TypeScript
- **StyleX** (`@stylexjs/stylex` + `unplugin-stylex/vite`) — compile-time atomic CSS
- **Vite** (library mode bundler — replaced tsup)
- `vite-plugin-dts` — TypeScript declaration file generation
- Storybook (dev/docs — local only, not published)
- Vitest (tests)
- `@floating-ui/react` — positioning + interaction hooks for anchored overlays; in use by `Dropdown` and `Tooltip`. `Popover` deferred (Modal may replace it). Marked `external` in `vite.config.ts` and listed under `dependencies` (not `devDependencies`) so it resolves transitively for consumers rather than bundling

## Design System: Cyber-Minimalist

- **Aesthetic**: Brutalist minimalism + terminal precision
- **Font**: Kode Mono (400–700 weight, loaded via Google Fonts)
- **Radius**: 0px everywhere (no exceptions)
- **Elevation**: tonal layering + 1px borders + a hard-edged, zero-blur offset shadow (`--void-shadow`) on raised/floating surfaces — never a soft/blurred shadow, that would read as glassmorphism and break the brutalist language
- **Accent**: violet (`#c160ef` / `#d175ff` hover) with glow rgba for focus rings
- **Border on focus/hover**: shifts from `--void-border` (#2a2a2a) to `--void-accent`
- **Button press ("keycap") mechanic**: `Button` sits raised above its shadow at rest, lifts slightly on `:hover` (`translate(-1px,-1px)` + shadow grows to 4px), and on `:active` translates exactly onto the shadow's offset (`translate(3px,3px)`) while the shadow collapses to `none` — the button visually bottoms out like a physical key being pressed flush, replacing the previous generic `scale(0.97)` press effect
